ТЕСТИРОВАНИЕ ПРОГРАММНОГО СРЕДСТВА (ПЛАТФОРМА ПОДАРОЧНЫХ СЕРТИФИКАТОВ)
================================================================================
Описание системы (контекст для тестов): веб-приложение на Laravel + Inertia/Vue
для выпуска и продажи подарочных сертификатов, ролей «администратор платформы»,
«бизнес», «менеджер точки», «клиент»; каталог сертификатов, заказы, оплата,
гашение (POS / менеджер), заявки на регистрацию бизнеса, организации, аналитика,
REST API (публичное, бизнес, админ).

2. МОДУЛЬНОЕ ТЕСТИРОВАНИЕ
--------------------------------------------------------------------------------
Цель: проверять изолированно отдельные единицы кода (классы, методы моделей,
простые сервисы) с подменой зависимостей или на in-memory БД, без полного HTTP-стека.

Подходы:
- PHPUnit: тесты в каталоге tests/Unit;
- для моделей Eloquent — фабрики + RefreshDatabase или транзакции;
- утверждения на исключения, возвращаемые значения, изменение атрибутов модели.

Ниже — примеры методов продукта с кратким описанием и полным текстом метода
(как в исходном коде приложения).

--- Метод 1: ClientTypeController::setType ---
Назначение: валидация типа клиента (client | business) и установка cookie
client_type на 30 суток; ответ JSON { ok: true }.

    public function setType(Request $request)
    {
        $validated = $request->validate([
            'clientType' => 'required|in:client,business'
        ]);

        return response()->json(['ok' => true])
            ->cookie('client_type', $validated['clientType'], 60 * 24 * 30); // 30 дней
    }

Модульные/юнит-сценарии вокруг этого метода (логика валидации Laravel):
- валидные значения client и business;
- невалидное значение — ожидание ValidationException / ответ 422 при вызове через HTTP
  (строго юнит-тест контроллера в Laravel чаще делают через feature-тест).

--- Метод 2: PurchasedCertificate::redeem ---
Назначение: списание суммы с купленного сертификата; проверка статуса active,
срока действия, достаточности баланса; при нулевом остатке — статус redeemed;
создание записи транзакции типа redeem.

    public function redeem(float $amount, string $description = null): bool
    {
        if ($this->status !== self::STATUS_ACTIVE) {
            throw new \RuntimeException('Certificate is not active');
        }

        if ($this->expires_at && $this->expires_at->isPast()) {
            $this->status = self::STATUS_EXPIRED;
            $this->save();
            throw new \RuntimeException('Certificate has expired');
        }

        if ($amount > $this->balance) {
            throw new \RuntimeException('Amount exceeds balance');
        }

        $this->balance -= $amount;

        if ($this->balance <= 0) {
            $this->status = self::STATUS_REDEEMED;
        }

        $this->save();

        // Создаем транзакцию
        $this->transactions()->create([
            'type' => 'redeem',
            'amount' => $amount,
            'description' => $description ?? 'Redeemed',
            'balance_after' => $this->balance,
        ]);

        return true;
    }

Примеры модульных тестов (имена методов тестов, смысл):
- test_redeem_decreases_balance_and_creates_transaction — успешное частичное списание;
- test_redeem_full_balance_sets_status_redeemed — полное списание;
- test_redeem_throws_when_not_active — статус не active;
- test_redeem_throws_when_expired — просроченный сертификат;
- test_redeem_throws_when_amount_exceeds_balance — сумма больше баланса.

--- Метод 3 (пример доменной логики): RegisterRequestController::accept ---
Назначение: принятие заявки бизнеса в транзакции БД — создание организации,
пользователя с ролью business, перенос хэша пароля из заявки, обновление статуса
заявки, отправка письма.

    public function accept(RegisterRequest $registerRequest): RedirectResponse
    {
        if ($registerRequest->status !== RegisterRequest::STATUS_PENDING) {
            return back()->withErrors(['status' => 'Заявка уже обработана.']);
        }

        DB::transaction(function () use ($registerRequest) {
            $slugBase = Str::slug($registerRequest->name);
            $slug = $slugBase.'-'.Str::lower(Str::random(6));

            $organization = Organization::create([
                'name' => $registerRequest->name,
                'slug' => $slug,
                'plan_name' => 'free',
                'subscription_active_until' => now()->addMonths(3),
            ]);

            $user = User::create([
                'name' => $registerRequest->contact,
                'email' => $registerRequest->email,
                'password' => Str::random(32),
                'organization_id' => $organization->id,
                'client_type' => 'business',
                'email_verified_at' => now(),
            ]);

            $hashed = DB::table('register_requests')
                ->where('id', $registerRequest->id)
                ->value('password');

            DB::table('users')->where('id', $user->id)->update(['password' => $hashed]);

            $user->assignRole('business');

            $registerRequest->update([
                'status' => RegisterRequest::STATUS_ACCEPTED,
                'organization_id' => $organization->id,
                'user_id' => $user->id,
                'reason' => null,
            ]);
            Mail::to($user)->send(new SendAcceptBusiness($registerRequest->email,$registerRequest));
        });

        return redirect()
            ->route('admin.register-requests.index')
            ->with('success', 'Заявка принята: создана организация и пользователь.');
    }

Для чистого модульного теста accept удобно вынести ядро в сервис и мокать Mail::fake();
в текущем виде чаще покрывают интеграционным feature-тестом.

3. ИНТЕГРАЦИОННОЕ ТЕСТИРОВАНИЕ
--------------------------------------------------------------------------------
Цель: проверять взаимодействие слоёв — HTTP, маршруты, middleware, БД, очереди,
внешние сервисы (с фейками). В Laravel это обычно tests/Feature с RefreshDatabase.

Границы интеграции для данного ПО:
- Auth + Sanctum для API бизнеса;
- middleware verified.business, subscription, role;
- контроллеры Inertia + редиректы;
- транзакции гашения (PurchasedCertificate / GiftCertificate + записи погашений);
- webhooks оплаты с подписью/валидацией payload.

Примеры уже существующих интеграционных тестов (имя метода, кратко, текст метода).

--- Feature: ClientTypeControllerTest::test_it_sets_client_type_cookie_for_valid_value ---
Смысл: POST /set-client-type с валидным телом устанавливает cookie и JSON ok.

    public function test_it_sets_client_type_cookie_for_valid_value(): void
    {
        $response = $this->postJson(route('set-client-type'), [
            'clientType' => 'client',
        ]);

        $response
            ->assertOk()
            ->assertJson(['ok' => true])
            ->assertCookie('client_type', 'client');
    }

--- Feature: ClientTypeControllerTest::test_it_returns_validation_error_for_invalid_client_type ---
Смысл: невалидный clientType даёт 422 и ошибку валидации.

    public function test_it_returns_validation_error_for_invalid_client_type(): void
    {
        $response = $this->postJson(route('set-client-type'), [
            'clientType' => 'invalid',
        ]);

        $response
            ->assertStatus(422)
            ->assertJsonValidationErrors(['clientType']);
    }
